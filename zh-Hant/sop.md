## §1 何時觸發

### 1.1 強制觸發條件 (T1-T4)

| 編號 | 觸發條件 | 說明 |
|------|---------|------|
| **T1** | 方法論資產產出 | 框架、SOP、分類體系、評估 rubric、模板——任何會被後續專案複用的結構性產出 |
| **T2** | 結論將進入正式交付物 | 論文、學術報告、對外發布的分析結論 |
| **T3** | 決策涉及資源分配或不可逆操作 | 資料刪除、架構重寫、公開發布、提交競賽結果 |
| **T4** | 兩個以上 AI/agent 給出衝突結論 | 需要獨立第三方裁決時 |

### 1.2 建議觸發條件 (S1-S3)

| 編號 | 觸發條件 |
|------|---------|
| **S1** | 單模型產出但複雜度高（超過 3 個推理步驟的鏈式結論） |
| **S2** | 使用了新穎/未經驗證的方法或工具鏈 |
| **S3** | 已知該模型在相關領域有系統性盲點 |

### 1.3 不觸發的情況

- 純格式轉換（無內容判斷）
- 已有獨立審查結論且輸入未變
- 使用者明確宣告"不需要審查"（需記錄宣告時間和上下文）。**限制**：T2/T3 不得靜默跳過，須記錄風險接受人、原因、時間戳。

**替代路徑**：當觸發 T1-T4 但裸環境條件（§5.1 C1-C4）無法滿足時（如僅有一個可用後端、無法開啟新會話等），審查不適用本 SOP 的完整四步流程。操作者須執行以下替代流程：
1. 記錄退化原因（為什麼 C1-C4 不可行）
2. 選擇最大可行的審查方式（如同後端交叉檢查、作者自審 + 記錄盲點風險）
3. 在報告中顯式宣告退化級別和風險（參照 §5.2）
4. 將報告標記為 `[NON]` 或 `[DEGRADED]`，不可作為正式獨立審查證據使用
  
替代流程產出的報告不是"獨立審查"，而是"受約束的內部交叉檢查"——其價值在於記錄已執行的驗證步驟和已知盲點，而非提供等效於 [IND] 的置信度。

## §2 獨立性定義與判定

### 2.1 兩軸判定模型

獨立性沿兩個軸判定，**缺一不可**：

**軸1: 後端模型獨立性**
- 審查者使用的 AI 模型必須與作者使用的模型**不同**
- "另一個 agent""另一個會話""同一模型的不同溫度引數"均不構成獨立
- 不同公司（Anthropic / OpenAI / DeepSeek / Google）→ 獨立
- 同一公司不同型號 → 半獨立（須在報告中宣告並降權）

**軸2: 上下文隔離性**
- 審查者不得接觸作者的中間推理、評分、結論
- 審查者只能接觸：源材料（被審查物件）+ 審查 rubric + 裸環境

### 2.2 獨立性三級分類

| 級別 | 軸1 | 軸2 | 標籤 | 用途 |
|------|-----|-----|------|------|
| **獨立 [IND]** | 不同後端 | 完全隔離 | `[IND]` | 可作為一級證據 |
| **半獨立 [SEMI]** | 同公司不同型號 或 軸2部分隔離 | `[SEMI]` | 二級證據，需宣告侷限。同一公司不同型號的獨立性取決於架構差異：不同架構族（如 GPT-4o vs o3）→ 可視為半獨立；同架構縮放（如 GPT-4o vs GPT-4o-mini）→ 偏 NON。具體判定由審查者在獨立性聲明中說明理由。 |
| **非獨立 [NON]** | 同一後端 或 軸2未隔離 | `[NON]` | 不可作為獨立審查使用 |

### 2.3 判定決策樹

```
審查者後端 ≠ 作者後端？
  ├─ 是 → 上下文完全隔離？
  │        ├─ 是 → [IND] 獨立
  │        └─ 否 → [SEMI] 半獨立（宣告洩露點）
  └─ 否 → 同公司不同型號？
           ├─ 是 → 上下文完全隔離？
           │        ├─ 是 → [SEMI] 半獨立
           │        └─ 否 → [NON] 非獨立
           └─ 否 → [NON] 非獨立
```

## §3 角色分工

### 3.1 三角色模型

| 角色 | 職責 | 不能是誰 |
|------|------|---------|
| **作者 (Author)** | 產出被審查內容 | — |
| **操作者 (Operator)** | 搭建裸環境、傳遞材料、確保資訊防火牆 | 不能是作者模型，不能是審查者模型 |
| **審查者 (Reviewer)** | 獨立評分 + 對抗挑戰 + 終判 | 不能與作者同後端，不能接觸作者中間產出 |

### 3.2 操作者的防火牆職責

- 從作者處提取源材料時，剝離所有評分、自評、結論性判斷
- 向審查者傳遞前，驗證裸環境完整性（§5 checklist）
- 不得向審查者透露"作者認為哪裡好/哪裡有問題"
- 不得向作者透露審查者的評分直到終判完成

## §4 執行流程

### 4.1 四步全景圖

```
源材料 → [步驟一: 事實提取] → [步驟二: 獨立評分] → [步驟三: 對抗式挑戰] → [步驟四: 終判]
              ↑                      ↑                    ↑                    ↑
         操作者傳遞              評分在挑戰前！         挑戰後調整評分         獨立性聲明嵌入
         (裸環境驗證)           (防汙染規則)           (adversarial turn)     (provenance 鎖定)
```

### 4.2 關鍵約束

1. **步驟二（評分）必須在步驟三（挑戰）之前**：評分基於初次理解，而非被挑戰塑造後的理解。顛倒將導致審查者被自己的挑戰帶偏。
2. **步驟三必須包含"魔鬼代言人不得軟化"約束**：挑戰方須盡最大努力找出問題，不得以"但總的來說還是不錯的"收尾。
3. **每步產出必須標註 provenance**：模型後端 + 會話標識 + 時間戳。

## §5 裸環境搭建

### 5.1 十項 Checklist

審查開始前，操作者逐項確認：

| 編號 | 檢查項 | 通過標準 |
|------|--------|---------|
| **C1** | 新會話 | 審查者會話不包含任何專案歷史 |
| **C2** | 無 memory 注入 | 審查者不載入使用者的 memory/偏好檔案 |
| **C3** | 無專案文件 | 只傳遞本 SOP + 源材料 + rubric |
| **C4** | 無作者評分 | 源材料中剝離所有自評/結論 |
| **C5** | 無中間產出 | 審查者不看到作者草稿/推理鏈 |
| **C6** | 不同後端 | 審查者使用與作者不同的模型後端 |
| **C7** | 無先驗標籤 | 不告知審查者"這是誰寫的""什麼水平" |
| **C8** | rubric 純淨 | 審查 rubric 不含導向性語言 |
| **C9** | 時間戳記錄 | 每步操作記錄時間戳 |
| **C10** | 版本綁定 | 被審材料版本標識已記錄（git hash/檔案 hash/版本號） |

### 5.2 退化檢測

- **完全裸環境**：C1-C10 全部通過 → [IND] 可達
- **部分退化**：C6 不可行（同後端）→ 最多 [NON]
- **嚴重退化**：C1-C4 任一不可行 → 降級為"內部交叉檢查"，不適用本 SOP

## §6 步驟一：事實提取

### 6.1 置信層級

| 層級 | 標籤 | 定義 |
|------|------|------|
| **L1** | `[VERIFIED]` | 可被公開可查來源驗證 |
| **L2** | `[CONSISTENT]` | 與審查者自身知識一致 |
| **L3** | `[PLAUSIBLE]` | 邏輯自洽但無法驗證 |
| **L4** | `[UNVERIFIABLE]` | 無法判斷真偽 |

> **編號說明**：L1-L4 為置信層級（Level）。觸發條件（§1.1）使用 T1-T4（Trigger）——兩套編號系統互不衝突。

### 6.2 聲明分類

每條提取的宣告標註型別：`[FACT]` 事實斷言 / `[INFERENCE]` 推導結論 / `[ASSUMPTION]` 未宣告假設 / `[VALUE]` 價值判斷。

### 6.3 工具受限標註

若審查者因工具限制無法驗證某宣告，標註 `[TOOL-LIMITED]` 而非降級置信度。

---

## §7 步驟二：獨立評分

### 7.1 六維評分框架

| 維度 | 名稱 | 範圍 | 說明 |
|------|------|------|------|
| **D1** | 邏輯有效性 | 1-5 | 推理鏈是否無邏輯跳躍/迴圈/矛盾 |
| **D2** | 事實準確性 | 1-5 | 可驗證事實是否正確 |
| **D3** | 方法適當性 | 1-5 | 所選方法是否適合研究問題 |
| **D4** | 完整性 | 1-5 | 是否覆蓋必要維度/邊界條件/替代解釋 |
| **D5** | 清晰度 | 1-5 | 表達是否清晰、結構是否合理 |
| **D6** | 洞察深度 | 1-5 | 是否有超出常規的洞察 |

### 7.2 評分約束

- **評推理不評結論（D2 例外）**：D1/D3-D6 評「推理如何到達結論」，不評「結論對不對」。**D2（事實準確性）例外**：先驗證事實斷言和結論可靠性，再評推理品質。D2 的核心是事實是否正確——這與結論可靠性直接相關，不適用"不評結論"約束。
- **N/A 允許**：某維度不適用時標記 N/A 並說明理由
- **錨定說明**：每項評分附一句話解釋
- **不知為不知**：缺乏領域知識時標記 `[OUT-OF-DOMAIN]`

### 7.3 綜合評分

綜合評分 = 各維度算術平均（N/A 維度排除）。不做加權——加權本身是價值判斷，應由人類使用者決定而非審查者代勞。**定製出口**：如果審查場景需要加權（如安全關鍵系統審查→D2 事實準確性加權），須在終判中宣告加權方案和理由。預設使用等權算術平均。

## §8 步驟三：對抗式挑戰

### 8.1 五方面必填挑戰

| 編號 | 挑戰 | 觸發問題 |
|------|------|---------|
| **(A)** | 替代解釋 | "有沒有其他解釋能同樣好地說明這些事實？" |
| **(B)** | 隱藏假設 | "哪些假設被當成了既定事實？" |
| **(C)** | 邊界條件 | "在什麼情況下這個結論會失效？" |
| **(D)** | 反例/反證 | "有沒有與結論矛盾的事實或案例？" |
| **(E)** | 方法論替代 | "用另一種方法會不會得出不同結論？" |

> **預設設定**：(A)-(E) 五個方面為全覆蓋挑戰，適用於方法論資產和正式交付物審查。**定製出口**：對於輕量審查場景（如日常程式碼片段、非正式內容），可選擇性省略某些方面——但須在終判中宣告省略的方面和理由。不得為"偷懶"而省略——如果省略 (A)-(E) 中的某個方面，必須是因為該方面對被審材料不適用，而非因為方便。

### 8.2 挑戰約束

1. **魔鬼代言人不得軟化**：每個挑戰以最尖銳形式提出，禁止附加緩和語
2. **具體而非抽象**：指向材料中的具體段落/宣告/推理步驟
3. **挑戰後評分調整**：完成 (A)-(E) 後重新審視步驟二評分

### 8.3 挑戰後評分調整規則

```
IF 挑戰發現了步驟二未注意到的問題：
  → 下調相關維度評分，記錄「從 X 調整為 Y，原因：挑戰 (Z) 揭示…」
IF 挑戰未發現新問題（材料經受住挑戰）：
  → 維持原評分，記錄「經受住 (A)-(E) 挑戰，評分不變」
IF 挑戰發現了致命問題：
  → 觸發紅旗 R1-R3（見 §11），可能直接建議「棄用」
```

## §9 步驟四：終判

### 9.1 四級終判

| 判決 | 含義 | 後續行動 |
|------|------|---------|
| **保留 (Keep)** | 材料合格，可直接使用 | 標註審查結論，歸檔 |
| **小改 (Minor)** | 有可改進之處但不影響核心 | 列出建議，無需重審 |
| **重大修改 (Major)** | 存在影響核心結論的問題 | 列出問題，修改後須重審 |
| **棄用 (Discard)** | 核心結論不可靠 | 記錄原因，觸發根因分析 |

### 9.2 終判輸出

1. 綜合評分（D1-D6 平均）+ 各維度明細
2. 獨立性級別（[IND] / [SEMI] / [NON]）
3. 關鍵挑戰摘要（A-E 中最重要的 2-3 個）
4. 終判結論 + 理由
5. 獨立性聲明（見 9.3）

### 9.3 獨立性聲明格式（強制嵌入）

```yaml
independence_declaration:
  reviewer_backend: "<模型名称>"
  author_backend: "<作者模型名称>"
  axis1_different_backend: true/false
  axis2_context_isolated: true/false
  independence_level: "IND | SEMI | NON"
  degradation_notes: "<退化说明，如无可省略>"
  session_id: "<审查者会话标识>"
  timestamp: "<ISO 8601>"
```

## §10 後置核驗

### 10.1 九項審計

| 編號 | 審計項 | 通過標準 |
|------|--------|---------|
| **A1** | 後端驗證 | 審查者後端 ≠ 作者後端 |
| **A2** | 會話隔離 | 審查者會話無專案上下文洩露 |
| **A3** | 評分在前 | 步驟二評分時間戳早於步驟三挑戰時間戳 |
| **A4** | 挑戰不軟化 | 步驟三 (A)-(E) 不包含緩和語 |
| **A5** | provenance 完整 | 每步有後端+會話+時間戳 |
| **A6** | 宣告嵌入 | 終判包含完整 independence_declaration |
| **A7** | 調整有據 | 評分調整記錄了幅度和理由 |
| **A8** | N/A 有理 | N/A 標記有合理說明 |
| **A9** | 版本綁定 | 審查物件版本標識可追溯 |

### 10.2 審計結論

| 結論 | 條件 | 後續 |
|------|------|------|
| **審計通過** | A1-A9 全部通過 | 可作為正式證據 |
| **有條件通過** | A5/A7/A8/A9 不通過但 A1-A4/A6 通過 | 標註條件使用 |
| **審計不通過** | A1-A4 或 A6 不通過 | 不可使用，須重審 |

## §11 紅旗條款

> **優先順序**：紅旗條款優先於後置核驗（§10）。當紅旗和審計項檢測到同一問題時（如 R5 換殼不換模 vs A1 後端驗證），按紅旗處理路徑執行（審查作廢/重審），不以審計降級替代。

以下任一觸發，審查暫停，由操作者決定後續：

| 編號 | 紅旗 | 建議處理 |
|------|------|---------|
| **R1** | 核心事實錯誤（L2 級以上，影響核心結論） | 建議棄用 |
| **R2** | 迴圈推理（結論被用作推導自身的前提） | 建議重大修改 |
| **R3** | 範疇錯誤（方法不適用於問題型別） | 建議棄用或重大修改 |
| **R4** | 獨立性偽證（審查者實際接觸了作者中間產出） | 審查作廢，重審 |
| **R5** | 換殼不換模（審查者和作者是同一後端的不同包裝） | 審查作廢 |
| **R6** | 審查衰減（評分/挑戰顯著低於該模型歷史表現） | 低置信度，考慮補充審查 |
| **R7** | 資料不可用替代（使用替代資料但未宣告侷限） | 標記資料汙染風險 |
| **R8** | 時間錯位（用過時資訊驗證當前結論） | 標記時間侷限 |
| **R9** | 裸環境洩露（C1-C9 未通過但未宣告） | 審查降級 |

## §12 常見失敗模式

### 12.1 偽獨立 (Pseudo-Independence)
**現象**："另一臺機器上的同一個模型"被當作"獨立審查者"  
**檢測**：軸1 後端相同 = 非獨立，無論包裝成什麼  
**預防**：軸1 強制不同後端，provenance 當場記錄

### 12.2 審查衰減 (Review Decay)
**現象**：審查者的對抗性隨時間遞減——第 5 次審查比第 1 次顯著溫和  
**檢測**：對比同一審查者在不同時間點的 (A)-(E) 挑戰尖銳度  
**預防**：魔鬼代言人不得軟化約束（§8.2），審計 A4

### 12.3 工具受限偽裝 (Tool-Limitation Camouflage)
**現象**：審查者因工具限制無法驗證，但未標註 [TOOL-LIMITED]  
**檢測**：逐條檢查步驟一宣告的驗證方式  
**預防**：§6.3 強制標註規則 + §7.2 的 N/A 和 [OUT-OF-DOMAIN] 機制

## §13 輸出格式

### 13.1 雙件規範

每次獨立審查產出兩份檔案：
1. **`.md` 檔案**（人讀）：完整審查報告，包含所有步驟、評分、挑戰、終判、審計結論
2. **`.json` 檔案**（機讀）：結構化資料，欄位與 md 章節一一對應

### 13.2 JSON 結構規範


## §14 術語表

| 術語 | 定義 |
|------|------|
| **後端 (Backend)** | 實際執行推理的模型，如 GPT-5.5, DeepSeek-V4-Pro, Qwen3.7-Max |
| **源材料 (Source Material)** | 被審查的產出物 |
| **裸環境 (Bare Environment)** | 不含專案上下文、不含作者中間產出、不含 memory 的審查會話（完整 checklist 見 §5.1 C1-C10） |
| **操作者 (Operator)** | 搭建裸環境並作為資訊防火牆的人/agent |
| **魔鬼代言人 (Devil's Advocate)** | 系統性地為對立面辯護，用於壓力測試 |
| **provenance** | 模型後端來源記錄，用於事後追溯獨立性 |
