## §1 何时触发

### 1.1 强制触发条件 (T1-T4)

| 编号 | 触发条件 | 说明 |
|------|---------|------|
| **T1** | 方法论资产产出 | 框架、SOP、分类体系、评估 rubric、模板——任何会被后续项目复用的结构性产出 |
| **T2** | 结论将进入正式交付物 | 论文、学术报告、对外发布的分析结论 |
| **T3** | 决策涉及资源分配或不可逆操作 | 数据删除、架构重写、公开发布、提交竞赛结果 |
| **T4** | 两个以上 AI/agent 给出冲突结论 | 需要独立第三方裁决时 |

### 1.2 建议触发条件 (S1-S3)

| 编号 | 触发条件 |
|------|---------|
| **S1** | 单模型产出但复杂度高（超过 3 个推理步骤的链式结论） |
| **S2** | 使用了新颖/未经验证的方法或工具链 |
| **S3** | 已知该模型在相关领域有系统性盲点 |

### 1.3 不触发的情况

- 纯格式转换（无内容判断）
- 已有独立审查结论且输入未变
- 用户明确声明"不需要审查"（需记录声明时间和上下文）。**限制**：T2/T3 不得静默跳过，须记录风险接受人、原因、时间戳。

## §2 独立性定义与判定

### 2.1 两轴判定模型

独立性沿两个轴判定，**缺一不可**：

**轴1: 后端模型独立性**
- 审查者使用的 AI 模型必须与作者使用的模型**不同**
- "另一个 agent""另一个会话""同一模型的不同温度参数"均不构成独立
- 不同公司（Anthropic / OpenAI / DeepSeek / Google）→ 独立
- 同一公司不同型号 → 半独立（须在报告中声明并降权）

**轴2: 上下文隔离性**
- 审查者不得接触作者的中间推理、评分、结论
- 审查者只能接触：源材料（被审查对象）+ 审查 rubric + 裸环境

### 2.2 独立性三级分类

| 级别 | 轴1 | 轴2 | 标签 | 用途 |
|------|-----|-----|------|------|
| **独立 [IND]** | 不同后端 | 完全隔离 | `[IND]` | 可作为一级证据 |
| **半独立 [SEMI]** | 同公司不同型号 或 轴2部分隔离 | `[SEMI]` | 二级证据，需声明局限 |
| **非独立 [NON]** | 同一后端 或 轴2未隔离 | `[NON]` | 不可作为独立审查使用 |

### 2.3 判定决策树

```
审查者后端 ≠ 作者后端？
  ├─ 是 → 上下文完全隔离？
  │        ├─ 是 → [IND] 独立
  │        └─ 否 → [SEMI] 半独立（声明泄露点）
  └─ 否 → 同公司不同型号？
           ├─ 是 → 上下文完全隔离？
           │        ├─ 是 → [SEMI] 半独立
           │        └─ 否 → [NON] 非独立
           └─ 否 → [NON] 非独立
```

## §3 角色分工

### 3.1 三角色模型

| 角色 | 职责 | 不能是谁 |
|------|------|---------|
| **作者 (Author)** | 产出被审查内容 | — |
| **操作者 (Operator)** | 搭建裸环境、传递材料、确保信息防火墙 | 不能是作者模型，不能是审查者模型 |
| **审查者 (Reviewer)** | 独立评分 + 对抗挑战 + 终判 | 不能与作者同后端，不能接触作者中间产出 |

### 3.2 操作者的防火墙职责

- 从作者处提取源材料时，剥离所有评分、自评、结论性判断
- 向审查者传递前，验证裸环境完整性（§5 checklist）
- 不得向审查者透露"作者认为哪里好/哪里有问题"
- 不得向作者透露审查者的评分直到终判完成

## §4 执行流程

### 4.1 四步全景图

```
源材料 → [步骤一: 事实提取] → [步骤二: 独立评分] → [步骤三: 对抗式挑战] → [步骤四: 终判]
              ↑                      ↑                    ↑                    ↑
         操作者传递              评分在挑战前！         挑战后调整评分         独立性声明嵌入
         (裸环境验证)           (防污染规则)           (adversarial turn)     (provenance 锁定)
```

### 4.2 关键约束

1. **步骤二（评分）必须在步骤三（挑战）之前**：评分基于初次理解，而非被挑战塑造后的理解。颠倒将导致审查者被自己的挑战带偏。
2. **步骤三必须包含"魔鬼代言人不得软化"约束**：挑战方须尽最大努力找出问题，不得以"但总的来说还是不错的"收尾。
3. **每步产出必须标注 provenance**：模型后端 + 会话标识 + 时间戳。

---

## §5 裸环境搭建

### 5.1 十项 Checklist

审查开始前，操作者逐项确认：

| 编号 | 检查项 | 通过标准 |
|------|--------|---------|
| **C1** | 新会话 | 审查者会话不包含任何项目历史 |
| **C2** | 无 memory 注入 | 审查者不加载用户的 memory/偏好文件 |
| **C3** | 无项目文档 | 只传递本 SOP + 源材料 + rubric |
| **C4** | 无作者评分 | 源材料中剥离所有自评/结论 |
| **C5** | 无中间产出 | 审查者不看到作者草稿/推理链 |
| **C6** | 不同后端 | 审查者使用与作者不同的模型后端 |
| **C7** | 无先验标签 | 不告知审查者"这是谁写的""什么水平" |
| **C8** | rubric 纯净 | 审查 rubric 不含导向性语言 |
| **C9** | 时间戳记录 | 每步操作记录时间戳 |
| **C10** | 版本绑定 | 被审材料版本标识已记录（git hash/文件 hash/版本号） |

### 5.2 退化检测

- **完全裸环境**：C1-C10 全部通过 → [IND] 可达
- **部分退化**：C6 不可行（同后端）→ 最多 [NON]
- **严重退化**：C1-C4 任一不可行 → 降级为"内部交叉检查"，不适用本 SOP

## §6 步骤一：事实提取

### 6.1 置信层级

| 层级 | 标签 | 定义 |
|------|------|------|
| **T1** | `[VERIFIED]` | 可被公开可查来源验证 |
| **T2** | `[CONSISTENT]` | 与审查者自身知识一致 |
| **T3** | `[PLAUSIBLE]` | 逻辑自洽但无法验证 |
| **T4** | `[UNVERIFIABLE]` | 无法判断真伪 |

### 6.2 声明分类

每条提取的声明标注类型：`[FACT]` 事实断言 / `[INFERENCE]` 推导结论 / `[ASSUMPTION]` 未声明假设 / `[VALUE]` 价值判断。

### 6.3 工具受限标注

若审查者因工具限制无法验证某声明，标注 `[TOOL-LIMITED]` 而非降级置信度。

---

## §7 步骤二：独立评分

### 7.1 六维评分框架

| 维度 | 名称 | 范围 | 说明 |
|------|------|------|------|
| **D1** | 逻辑有效性 | 1-5 | 推理链是否无逻辑跳跃/循环/矛盾 |
| **D2** | 事实准确性 | 1-5 | 可验证事实是否正确 |
| **D3** | 方法适当性 | 1-5 | 所选方法是否适合研究问题 |
| **D4** | 完整性 | 1-5 | 是否覆盖必要维度/边界条件/替代解释 |
| **D5** | 清晰度 | 1-5 | 表达是否清晰、结构是否合理 |
| **D6** | 洞察深度 | 1-5 | 是否有超出常规的洞察 |

### 7.2 评分约束

- **D2 例外**：先验证事实和结论可靠性，再评推理质量
- **N/A 允许**：某维度不适用时标记 N/A 并说明理由
- **锚定说明**：每项评分附一句话解释
- **不知为不知**：缺乏领域知识时标记 `[OUT-OF-DOMAIN]`

### 7.3 综合评分

综合评分 = 各维度算术平均（N/A 维度排除）。不做加权——加权本身是价值判断。

---

## §8 步骤三：对抗式挑战

### 8.1 五方面必填挑战

| 编号 | 挑战 | 触发问题 |
|------|------|---------|
| **(A)** | 替代解释 | "有没有其他解释能同样好地说明这些事实？" |
| **(B)** | 隐藏假设 | "哪些假设被当成了既定事实？" |
| **(C)** | 边界条件 | "在什么情况下这个结论会失效？" |
| **(D)** | 反例/反证 | "有没有与结论矛盾的事实或案例？" |
| **(E)** | 方法论替代 | "用另一种方法会不会得出不同结论？" |

### 8.2 挑战约束

1. **魔鬼代言人不得软化**：每个挑战以最尖锐形式提出，禁止附加缓和语
2. **具体而非抽象**：指向材料中的具体段落/声明/推理步骤
3. **挑战后评分调整**：完成 (A)-(E) 后重新审视步骤二评分

---

## §9 步骤四：终判

### 9.1 四级终判

| 判决 | 含义 | 后续行动 |
|------|------|---------|
| **保留 (Keep)** | 材料合格，可直接使用 | 标注审查结论，归档 |
| **小改 (Minor)** | 有可改进之处但不影响核心 | 列出建议，无需重审 |
| **重大修改 (Major)** | 存在影响核心结论的问题 | 列出问题，修改后须重审 |
| **弃用 (Discard)** | 核心结论不可靠 | 记录原因，触发根因分析 |

### 9.2 终判输出

1. 综合评分（D1-D6 平均）+ 各维度明细
2. 独立性级别（[IND] / [SEMI] / [NON]）
3. 关键挑战摘要（A-E 中最重要的 2-3 个）
4. 终判结论 + 理由
5. 独立性声明（见 9.3）

### 9.3 独立性声明格式（强制嵌入）

```yaml
independence_declaration:
  reviewer_backend: "<模型名称>"
  author_backend: "<作者模型名称>"
  axis1_different_backend: true/false
  axis2_context_isolated: true/false
  independence_level: "IND | SEMI | NON"
  degradation_notes: "<退化说明，如无可省略>"
  session_id: "<审查者会话标识>"
  timestamp: "<ISO 8601>"
```

## §10 后置核验

### 10.1 九项审计

| 编号 | 审计项 | 通过标准 |
|------|--------|---------|
| **A1** | 后端验证 | 审查者后端 ≠ 作者后端 |
| **A2** | 会话隔离 | 审查者会话无项目上下文泄露 |
| **A3** | 评分在前 | 步骤二评分时间戳早于步骤三挑战时间戳 |
| **A4** | 挑战不软化 | 步骤三 (A)-(E) 不包含缓和语 |
| **A5** | provenance 完整 | 每步有后端+会话+时间戳 |
| **A6** | 声明嵌入 | 终判包含完整 independence_declaration |
| **A7** | 调整有据 | 评分调整记录了幅度和理由 |
| **A8** | N/A 有理 | N/A 标记有合理说明 |
| **A9** | 版本绑定 | 审查对象版本标识可追溯 |

### 10.2 审计结论

| 结论 | 条件 | 后续 |
|------|------|------|
| **审计通过** | A1-A9 全部通过 | 可作为正式证据 |
| **有条件通过** | A5/A7/A8/A9 不通过但 A1-A4/A6 通过 | 标注条件使用 |
| **审计不通过** | A1-A4 或 A6 不通过 | 不可使用，须重审 |

## §11 红旗条款

以下任一触发，审查暂停，由操作者决定后续：

| 编号 | 红旗 | 建议处理 |
|------|------|---------|
| **R1** | 核心事实错误（T2 级以上，影响核心结论） | 建议弃用 |
| **R2** | 循环推理（结论被用作推导自身的前提） | 建议重大修改 |
| **R3** | 范畴错误（方法不适用于问题类型） | 建议弃用或重大修改 |
| **R4** | 独立性伪证（审查者实际接触了作者中间产出） | 审查作废，重审 |
| **R5** | 换壳不换模（审查者和作者是同一后端的不同包装） | 审查作废 |
| **R6** | 审查衰减（评分/挑战显著低于该模型历史表现） | 低置信度，考虑补充审查 |
| **R7** | 数据不可用替代（使用替代数据但未声明局限） | 标记数据污染风险 |
| **R8** | 时间错位（用过时信息验证当前结论） | 标记时间局限 |
| **R9** | 裸环境泄露（C1-C9 未通过但未声明） | 审查降级 |

## §12 常见失败模式

### 12.1 伪独立 (Pseudo-Independence)
**现象**："另一台机器上的同一个模型"被当作"独立审查者"  
**检测**：轴1 后端相同 = 非独立，无论包装成什么  
**预防**：轴1 强制不同后端，provenance 当场记录

### 12.2 审查衰减 (Review Decay)
**现象**：审查者的对抗性随时间递减——第 5 次审查比第 1 次显著温和  
**检测**：对比同一审查者在不同时间点的 (A)-(E) 挑战尖锐度  
**预防**：魔鬼代言人不得软化约束（§8.2），审计 A4

### 12.3 工具受限伪装 (Tool-Limitation Camouflage)
**现象**：审查者因工具限制无法验证，但未标注 [TOOL-LIMITED]  
**检测**：逐条检查步骤一声明的验证方式  
**预防**：§6.3 强制标注规则 + §7.2 的 N/A 和 [OUT-OF-DOMAIN] 机制

## §13 输出格式

每次独立审查产出两份文件：
1. **`.md` 文件**（人读）：完整审查报告
2. **`.json` 文件**（机读）：结构化数据，字段与 md 一一对应

详细 JSON schema 见步骤四输出规范（§9.2）。

## §14 术语表

| 术语 | 定义 |
|------|------|
| **后端 (Backend)** | 实际执行推理的模型，如 GPT-5.5, DeepSeek-V4-Pro, Qwen3.7-Max |
| **源材料 (Source Material)** | 被审查的产出物 |
| **裸环境 (Bare Environment)** | 不含项目上下文、不含作者中间产出、不含 memory 的审查会话 |
| **操作者 (Operator)** | 搭建裸环境并作为信息防火墙的人/agent |
| **魔鬼代言人 (Devil's Advocate)** | 系统性地为对立面辩护，用于压力测试 |
| **provenance** | 模型后端来源记录，用于事后追溯独立性 |
