## D1: SOP 逻辑自洽性

**整体评价**：四步流程逻辑连贯，步骤间的前后依赖明确（事实→评分→挑战→终判）。但在约束优先级和异常处理路径上存在 2 个结构性缺口。

**具体发现**：

1. **[MAJOR] §5.2 退化检测与 §1.3 不触发的冲突**
   - `sop.md §1.3` 规定"T2/T3 不得静默跳过，须记录风险接受人、原因、时间戳"。但 `§5.2` 说 C6 不可行时降级为 `[NON]`，C1-C4 不可行时降级为"内部交叉检查，不适用本 SOP"。
   - **裂缝**：如果 T2 级材料遇到 C1 不可行（无法开新会话），SOP 给出的路径是"不适用本 SOP"——但 §1.3 同时说 T2 不得跳过。操作者面对的是一个**逻辑死锁**：既不能跳过审查，又被告知不适用本 SOP。缺少"在 SOP 不适用时如何替代"的指引。

2. **[MODERATE] §11 红旗条款 R4/R5 与 §10 后置核验的管辖权重叠**
   - `§10.1 A1`（后端验证）和 `§11 R5`（换壳不换模）检测的是同一类问题（后端是否真不同），但处理路径不同：A1 不通过→审计不通过→不可使用；R5 触发→审查作废→重审。
   - **裂缝**：两者的语义差异（"不可使用" vs "作废+重审"）在实操上是否相同？操作者应在何时引用哪一条？缺乏优先级说明。

3. **[MINOR] §8.3 挑战后评分调整规则与 §10.1 A3 审计项的张力**
   - `§4.2` 强制"评分必须在挑战之前"，`A3` 审计"步骤二时间戳早于步骤三"。但 `§8.3` 允许挑战后调整评分——调整后的评分时间戳算什么？如果调整发生在挑战之后，严格说 A3 仍然通过（初始评分在前），但如果审查者把"调整后评分"当成最终评分写入终判，语义上"评分在挑战之前"的精神已被部分架空。

**改进建议**：
- §1.3 增加"替代路径"：当 SOP 不适用时，操作者应记录"使用替代审查机制 + 理由 + 风险接受人"。
- §11 增加一条注释："红旗条款优先于后置核验——当红旗和审计项检测到同一问题时，红旗处理路径生效。"
- §8.3 明确："初始评分时间戳为 A3 审计锚点。挑战后调整不改变初始评分的时间戳记录。"

## D2: 概念定义质量

**整体评价**：核心概念定义总体清晰，双轴独立性模型和置信层级是 SOP 最强的设计元素。但存在 2 处跨节定义不一致。

**具体发现**：

1. **[MAJOR] "证据层级"标签命名冲突**
   - `§6.1` 定义置信层级标签为 `T1/T2/T3/T4`（VERIFIED/CONSISTENT/PLAUSIBLE/UNVERIFIABLE）。
   - `§1.1` 定义强制触发条件也用 `T1/T2/T3/T4`。
   - 同一个 SOP 中 `T1` 既是"可被公开验证的声明"又是"方法论资产产出"触发条件。`T2` 既是"与审查者知识一致"又是"结论将进入正式交付物"。这会造成**严重的认知混淆**——尤其是 AI 审查者解析 prompt 时可能将两套 T 编号混为一谈。

2. **[MODERATE] "裸环境"定义在 §5 和 §14 术语表中粒度不一致**
   - `§14` 术语表："不含项目上下文、不含作者中间产出、不含 memory 的审查会话"——3 项要素。
   - `§5.1` checklist：10 项（C1-C10），涵盖新会话、无 memory、无项目文档、无作者评分、无中间产出、不同后端、无先验标签、rubric 纯净、时间戳、版本绑定。
   - **裂缝**：术语表是高度简化的摘要。首次接触的读者如果只看 §14，会以为"裸环境"只需要满足 3 项条件。而 §5.1 实际操作需要 10 项。术语表作为概念入口应至少提示"详见 §5 checklist"。

3. **[MODERATE] "半独立 [SEMI]"的判定条件模糊**
   - `§2.2` 表格中，SEMI 的轴1 条件是"同公司不同型号 **或** 轴2部分隔离"。但"同公司不同型号"是一个连续光谱（GPT-4o vs GPT-5.5 差异远大于 Claude Sonnet 4 vs Claude Opus 4）。SOP 未给出判定"型号差异是否足够"的标准，导致 SEMI 和 IND 的边界由操作者主观判断——这与 SOP 追求客观性的精神矛盾。

**改进建议**：
- 将触发条件从 `T1-T4` 重命名为 `TC1-TC4`（Trigger Condition）或保留 `T1-T4` 但把置信层级改为 `L1-L4`（Level），消除命名冲突。这是整个 SOP 最紧急的修复。
- §14 术语表的"裸环境"条目末尾加 `→ 详见 §5 checklist`。
- §2.2 SEMI 条件增加注释："同公司不同型号的独立性取决于架构差异（如 GPT-4o vs o3 = 不同架构族 → 可视为半独立；GPT-4o vs GPT-4o-mini = 同架构缩放 → 偏 NON）。具体判定由审查者在独立性声明中论证。"

## D3: Prompt 模板可用性

**整体评价**：两个 prompt 模板结构清晰、自包含性好。full-review 的四步指令足够明确。但存在 1 个可操作性问题和 1 个边界模糊问题。

**具体发现**：

1. **[MAJOR] full-review.md 步骤四要求"双件输出"但缺乏 JSON schema**
   - `prompts/full-review.md` 步骤四第 5 项："同时输出一份结构化 JSON（字段与上述报告一一对应，格式参照 SOP §13.2）"。
   - 但 prompt 模板的设计意图是**复制到全新会话**——审查者 AI 不一定能访问 SOP。如果操作者只粘贴了 prompt + 材料，审查者看到的"格式参照 SOP §13.2"是一个**死引用**——它指向审查者不可能看到的外部文档。
   - **后果**：审查者 AI 会自行猜测 JSON 格式，导致输出不一致，破坏双件规范的可比性。

2. **[MODERATE] full-review 和 adversarial 的使用边界未在 prompt 自身中声明**
   - `README.md` 的场景表区分了两者，但 prompt 文件本身缺少"何时用我/何时不用我"的元信息。
   - adversarial.md 开头有一句"用于只做压力测试的场景"，但没有说明"如果你需要完整审查，请使用 full-review.md"。
   - full-review.md 没有说明"如果你只需要对抗挑战，请使用 adversarial.md"。
   - **风险**：操作者在没有 README 参考的情况下（比如从 `_reviews/prompts/` 目录直接取用），无法判断该选哪个模板。

3. **[MINOR] full-review.md 独立性声明 YAML 中 `session_id` 的含义对 AI 审查者不明确**
   - `session_id: "<审查者会话标识>"` — AI 审查者不一定知道自己所在平台的"会话标识"是什么（ChatGPT 没有暴露 session ID 给模型本身）。缺少 fallback 指引（如"如无平台级 session ID，记录会话开始时间 + 模型名"）。

**改进建议**：
- 将 SOP §13.2 的 JSON schema 直接内联到 full-review.md 的步骤四中（或至少提供一个最小化 schema），消除外部依赖。
- 两个 prompt 文件开头各加一行 `> 适用场景：…` / `> 不适用场景：…`。
- `session_id` 字段增加 fallback：`"<平台会话 ID，如无则用模型名+开始时间>"`。

## D4: 与源材料的一致性

**整体评价**：无法完全验证——我没有访问源框架 v1.6.4 的权限。基于仓库内可交叉推断的信息，提取转换基本忠实，但存在 1 个值得关注的语义漂移。

**具体发现**：

1. **[MODERATE] 从"描述性"到"规定性"的转换丢失了上下文判断的弹性**
   - 源框架的 SOP 是"记录已实践的流程"——它描述的是在特定项目上下文中**实际执行过**的步骤。独立版 SOP 将其转为"指导怎么操作"的规定性文档。
   - **漂移点**：描述性文档中，读者能自行判断"这一步在我的场景中是否需要"，因为原文有具体项目上下文作为参照。规定性文档中，每一步都变成了**规则**——但 SOP 并未为每个规则提供"何时可以合理变通"的指引。
   - 例：`§7.3` "综合评分 = 算术平均，不做加权"——在源框架的特定项目上下文中这是合理的默认选择。但作为通用 SOP，某些审查场景（如安全关键系统审查）可能需要 D2 事实准确性加权。SOP 缺少"权重可定制"的出口。

2. **[MINOR] 框架特定术语的残留**
   - `examples/methodology-review.md` 中引用了"[E-] 证据标注体系"和"Tier 0"等框架特有概念。对于不使用源框架的独立用户，这些术语是黑箱。示例文件应在使用这些术语时提供一句话解释。

**改进建议**：
- 对关键约束（如 §7.3 不做加权、§8.1 五方面必填）增加一行注释："这是默认设置。如需调整（如加权、省略某方面），须在终判中声明并说明理由。"
- 示例文件中对框架特有术语加内联解释或脚注。

## D5: 整体印象

**整体评价**：这是一个**超出平均水平的 AI 方法论工具包**——结构完整、概念原创性强、有真实案例支撑。README 的语气在自信和诚实之间取得了良好平衡（验证状态框尤其出色）。

**具体发现**：

1. **[MODERATE] 首次用户的认知负荷过高**
   - README 的"5 分钟上手"部分承诺了快速启动，但第一步"理解独立性"需要理解双轴模型，第二步需要区分 full-review vs adversarial，第三步涉及 10 项 checklist。实际认知负荷远超 5 分钟。
   - 对比：如果 README 提供一个**最简版 prompt**（比如一个 30 行的精简版 full-review），新用户可以在 2 分钟内复制粘贴并开始。当前路径要求用户先理解概念体系再行动。

2. **[MODERATE] 示例文件的定位模糊**
   - `examples/methodology-review.md` 自称"教学重构示例"——但它在目录结构中的位置暗示它是"真实案例"。读者点进去会发现大量框架特有引用（§6.3.2、Tier 0、[E-]、prompt-tdd 项目），这对独立用户来说**信息噪声大于教学价值**。
   - 缺少一个**自包含的**、不依赖源框架知识的简单示例（如审查一份 AI 写的技术博客文章）。

3. **[MINOR] 实战数据表的可验证性**
   - README 中的实战数据表（50+ 轮、5 种后端、~50% 互补率）没有链接到原始数据。作为独立产品的可信度声明，这些数据要么可验证（链接到源仓库具体章节），要么应该标注"自报告"。

**改进建议**：
- 在 README 的"5 分钟上手"之前增加一个"极速版"折叠块或链接：一个 20-30 行的最小化审查 prompt，供想快速试用的用户。
- 补充一个不依赖源框架的简单示例（如"审查一份 AI 生成的竞品分析报告"），降低示例的认知门槛。
- 实战数据表中每一项增加来源链接或标注"自报告"。

## 总体终判

| 维度 | 评价 | 发现数 |
|------|------|--------|
| D1 SOP 逻辑自洽性 | 主体坚实，有 2 个结构性缺口 | 1 MAJOR + 1 MODERATE + 1 MINOR |
| D2 概念定义质量 | 核心设计优秀，但有命名冲突 | 1 MAJOR + 2 MODERATE |
| D3 Prompt 模板可用性 | 自包含性好，有 1 个死引用 | 1 MAJOR + 1 MODERATE + 1 MINOR |
| D4 源材料一致性 | 基本忠实，有 1 处语义漂移 | 1 MODERATE + 1 MINOR |
| D5 整体印象 | 超出平均水平，认知负荷偏高 | 2 MODERATE + 1 MINOR |

### 终判：**Major Revision**

**理由**：

3 项 MAJOR 级发现需要修复后才能投入使用：

1. **T1-T4 命名冲突**（D2）——同一 SOP 中两套编号系统碰撞，对 AI 审查者构成解析歧义，对人类读者构成认知混淆。这是**最高优先级修复项**。
2. **§1.3 + §5.2 逻辑死锁**（D1）——T2 材料遇到 C1-C4 不可行时无解，操作者陷入"不能跳过"与"不适用"的矛盾。
3. **full-review.md 步骤四的 JSON schema 死引用**（D3）——prompt 在新会话中使用时，审查者无法访问 SOP §13.2，导致双件输出格式不一致。

这三项都是**结构性问题**——不是文字错误或遗漏，而是概念设计层面的裂缝。修复工作量不大（预估 2-4 小时），但修复前会影响 SOP 的可执行性和 prompt 的可靠性。

修复后预期评级：**Minor Revision**——剩余 MODERATE 级发现属于增强性改进，不阻碍基本使用。
